<p>&lt;p&gt;&lt;strong&gt;Architectural Changes&lt;/strong&gt;&lt;/p&gt;</p>
<p>&lt;p&gt;&lt;strong&gt; &lt;/strong&gt;There are major and minor architectural changes in the product since 9.0.1, let us discuss about them in detail:&lt;/p&gt;</p>
<p>&lt;ul&gt;</p>
<p>&lt;li&gt;&lt;strong&gt;Major change&lt;/strong&gt;: In 9.1 and earlier versions, the consolidation logic is resided in database and any change in the core functionality is specific to the database. To move away from the database dependency, there is a major release (9.6.x) which is known as database agnostic release. With this release, the consolidation logic is moved over to Java and the process is run on application server for better performance. Also, we started supporting Microsoft SQL Server database with this architecture change.&lt;/li&gt;</p>
<p>&lt;li&gt;&lt;strong&gt;Minor change (optional)&lt;/strong&gt;: &amp;nbsp;If your MDM implementation is looking forward to use Informatica platform, then there is a change in the architecture to integrate Platform to MDM.&lt;/li&gt;</p>
<p>&lt;/ul&gt;</p>
<p>&lt;p&gt;&lt;img src=&quot;/_layouts/InformaticaKBMasterPage/Images/ea/MDM Platform Integration.png&quot; alt=&quot;&quot; width=&quot;100%&quot; /&gt;&lt;/p&gt;</p>
<p>&lt;p&gt;&lt;br /&gt;With this integration, there is a change in the data flow within Hub.&lt;/p&gt;</p>
<p>&lt;p&gt;&amp;nbsp;&lt;/p&gt;</p>
<p>&lt;p&gt;&lt;strong&gt;&lt;em&gt;Traditional Flow&lt;/em&gt;&lt;/strong&gt;&lt;/p&gt;</p>
<p>&lt;p&gt;ETL process ends by populating the Landing table. MDM jobs move the data from Landing to Staging through Stage process and then to Base Objects through Load process.&lt;/p&gt;</p>
<p>&lt;p&gt;&amp;nbsp;&lt;/p&gt;</p>
<p>&lt;p&gt;&lt;strong&gt;&lt;em&gt;Data flow with Platform Integration&lt;/em&gt;&lt;/strong&gt;&lt;/p&gt;</p>
<p>&lt;p&gt;After integrating the MDM Hub with the Informatica platform, the Data Integration Service loads source data directly into MDM Hub staging tables&amp;nbsp;by eliminating the step of loading to Landing table.&amp;nbsp;The user uses Informatica Developer tool to create mappings with transformations that are executed by the Data Integration Service&lt;/p&gt;</p>

<p>&lt;p&gt;&amp;nbsp;&lt;/p&gt;</p>
<p>&lt;p&gt;&lt;strong&gt;&lt;em&gt;Traditional Flow&lt;/em&gt;&lt;/strong&gt;&lt;/p&gt;</p>
<p>&lt;p&gt;ETL process ends by populating the Landing table. MDM jobs move the data from Landing to Staging through Stage process and then to Base objects through Load process.&lt;/p&gt;</p>
<p>&lt;p&gt;&amp;nbsp;&lt;/p&gt;</p>
<p>&lt;p&gt;&lt;em&gt;&lt;strong&gt;Data flow with Platform Integration&lt;/strong&gt;&lt;/em&gt;&lt;/p&gt;</p>
<p>&lt;p&gt;After integrating the MDM Hub with the Informatica platform, the Data Integration Service loads source data directly into MDM Hub staging tables&amp;nbsp;by eliminating the step of loading to Landing table.&amp;nbsp;The user uses Informatica Developer tool to create mappings with transformations that are executed by the Data Integration Service&lt;/p&gt;</p>

<p>&lt;li&gt;&lt;strong&gt;Raw retention&lt;/strong&gt;: This deprecation is applicable only for Platform integration enabled Staging tables. As the data flow to staging table is done outside MDM, it is outside the scope of MDM to get the RAW data and has to be taken care in the ETL process itself.&lt;/li&gt;</p>
<p>&lt;li&gt;&lt;strong&gt;Native IDD workflow engine&lt;/strong&gt;: In earlier versions, Siperian BPM engine is the default workflow engine in IDD. From v10.0, it is recommended to install ActiveVOS BPM engine during Hub server installation and AVOS will be the default workflow engine in IDD.&lt;/li&gt;</p>

<p>&lt;li&gt;&lt;strong&gt;SIF APIs&lt;/strong&gt;: With the introduction to timeline functionality, most of the API's are added with the new elements for specifying effective dates.&lt;/li&gt;</p>
<p>&lt;li&gt;&lt;strong&gt;User exit support&lt;/strong&gt;: DB user exits are no more supported, all the custom code has to be moved to Java user exits.&lt;/li&gt;</p>
<p>&lt;li&gt;&lt;strong&gt;Default Proxy role&lt;/strong&gt;: From v10.0, creating a new ORS will not create the default proxy role. In the case of upgrade, existing proxy role is converted to an ordinary role which can be deleted.&lt;/li&gt;</p>

<p>&lt;ul&gt;</p>
